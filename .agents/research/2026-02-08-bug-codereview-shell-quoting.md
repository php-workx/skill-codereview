# Bug Report: Codereview Tool Runner Shell Quoting Failure

**Date:** 2026-02-08
**Severity:** medium
**Status:** root-cause-found

## Symptom
Skill execution failed during deterministic scan status collection with shell parse errors, including:
- `zsh: bad pattern`
- `zsh: no matches found: .Vulnerabilities?`
- `zsh: no such file or directory: vulnerabilities ...`

## Expected Behavior
Deterministic scanners should run and emit `TOOL_STATUS_*` lines without shell parse errors.

## Root Cause
- **What:** Multiline script content was wrapped inside `bash -lc '...'` while the inner script also used single-quoted jq/rg patterns.
- **Where:** Runner command construction (ad-hoc tool execution string), not scanner logic.
- **Why:** Inner single quotes ended the outer single-quoted string early; the parent `zsh` shell then interpreted fragments (`?`, `|`) as glob/pattern tokens.
- **When introduced:** During command assembly in session `2026-02-08T13-11-10`.

## Working Pattern Comparison
- **Broken pattern:** `bash -lc '... jq '.results | length' ...'`
- **Working pattern:** write real script via `<<'EOF'` heredoc and execute script file with `bash`.

## Hypothesis Tested
Hypothesis: parse failures are caused by quote nesting, not scanner binaries.
- Test: run same logic from `/tmp/codereview_run_tools.sh` generated by quoted heredoc.
- Result: script executed successfully and produced `TOOL_STATUS_BEGIN ... TOOL_STATUS_END`.

## Fix Applied
Updated `skill/SKILL.md` Step 3 to add explicit execution-safety guidance:
- avoid multiline `bash -lc '...'` with inner single quotes
- use heredoc script file execution pattern

## Verification
- Confirmed failure signature in prior session log.
- Confirmed safe-script pattern runs and returns tool status lines.
